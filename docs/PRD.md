# Wusaki Agent PRD

## 1. 背景

`wusaki-agent` 计划吸收 `akashic-agent` 已验证的核心能力，从“只会响应消息的 Agent”升级为“既能被动对话，也能主动陪伴，并且具备长期记忆与空闲自驱能力的个人 Agent”。

本 PRD 聚焦第一阶段产品定义，优先借鉴 `akashic-agent` 中四类主能力：

1. 被动回复
2. 主动推送
3. 记忆系统
4. Drift 空闲任务

目标不是 1:1 复制 `akashic-agent` 的全部实现，而是为 `wusaki-agent` 明确一套可落地、可分期、可验收的产品范围。

## 2. 产品目标

### 2.1 产品愿景

让 `wusaki-agent` 成为一个“持续在线、理解上下文、会主动行动”的陪伴型 Agent，而不是一次性问答工具。

### 2.2 一期目标

一期目标是完成以下能力闭环：

1. 用户发来消息时，Agent 能结合近期上下文、长期记忆和工具能力，给出稳定的被动回复。
2. 用户不发消息时，Agent 能基于订阅源和规则判断是否需要主动推送内容。
3. 对话产生的信息能沉淀为可检索、可归档、可纠错的记忆。
4. 在没有主动推送任务时，Agent 能执行受控的空闲任务，持续改善记忆质量或推进轻量后台工作。

### 2.3 成功标准

产品层面的成功标准：

1. `wusaki-agent` 能同时跑通“被动回复 + 记忆写入/召回 + 主动推送 + Drift”四条主链路。
2. 主动能力可配置、可关闭、可审计，不会变成不可控打扰源。
3. 记忆内容对回复和推送决策有实际帮助，而不是只做存储。
4. Drift 任务可以通过文档化任务定义扩展，而不要求每次都改核心代码。

## 2.4 已确定产品决策

以下关键方案在一期内视为已确定，不再作为待选项：

1. 通信渠道一期支持 `QQ Bot` 和 `Telegram Bot`，并为后续接入 `Feishu Bot` 预留统一渠道抽象。
2. 向量记忆沿用 `akashic-agent` 的双层记忆思路与实现路线，即 Markdown 记忆层 + 向量检索层。
3. 主动推送的数据源抽象沿用 `MCP`，不另起新的接入协议。
4. Drift 任务定义直接复用 `SKILL.md` 规范。
5. 一期纳入最小化的开发调试 `CLI`，用于本地运行、排查和验证主链路。

## 3. 目标用户与核心场景

### 3.1 目标用户

一期默认面向以下用户：

1. 需要长期陪伴和上下文连续性的个人用户
2. 希望 Agent 帮助跟进信息源、提醒事项、兴趣主题的重度用户
3. 愿意维护少量规则和配置，以换取更高自主性的高级用户

### 3.2 核心场景

1. 用户和 Agent 日常聊天，Agent 能记住偏好、项目、计划和近期状态。
2. 用户订阅特定信息源后，Agent 能在合适时机主动推送值得关注的内容。
3. 用户要求“记住这件事”或“你记错了”后，系统能更新长期记忆。
4. 空闲时 Agent 能做轻量后台工作，例如记忆审计、用户画像补全、待办梳理、自检等。

## 4. 产品原则

1. 可控优先：主动消息必须可限制频率、可配置来源、可显式关闭。
2. 记忆可信：记忆要支持来源追踪、纠错和去重，避免“越记越错”。
3. 上下文连续：被动回复、主动推送和 Drift 应共享同一套用户上下文基础设施。
4. 渐进式演进：先做核心闭环，再扩展复杂插件、可视化面板和高级策略。
5. 可观测：关键决策要能被记录，便于排查“为什么发了/没发”“为什么记住了/忘了”。

## 5. 功能范围

## 5.1 一期范围总览

一期纳入以下功能模块：

1. 被动回复主链路
2. 主动推送主链路
3. Markdown + 向量双层记忆
4. Drift 空闲任务框架
5. 多渠道接入能力
6. 最小化开发调试 CLI
7. 基础配置、调度、日志与审计能力

一期暂不要求完整复刻 `akashic-agent` 的所有插件生态、Dashboard UI 和复杂调试工具，但要为后续扩展留出边界。

## 5.2 模块一：被动回复

### 功能目标

用户发来消息后，Agent 能完成一轮稳定响应，支持上下文注入、记忆召回、工具调用和结果输出。

### 范围定义

1. 接收用户消息并创建一轮会话处理流程。
2. 在回复前注入系统设定、长期记忆、近期上下文和必要检索结果。
3. 支持最小可用工具集，例如记忆写入、记忆检索、网页获取、消息发送类工具。
4. 输出最终回复，并将本轮对话提交给后处理链路。
5. 为后续扩展保留阶段化生命周期或中间件挂载点。
6. 一期至少在 `QQ Bot` 和 `Telegram Bot` 上打通同一套被动回复链路。

### 借鉴 Akashic 的点

1. 回复前进行记忆召回，而不是只依赖短窗口上下文。
2. 一轮对话结束后立即进入记忆维护链路。
3. 工具能力与主对话循环解耦。

### 一期不包含

1. 完整插件市场
2. 复杂多通道适配
3. 高度细化的生命周期事件总线

## 5.3 模块二：主动推送

### 功能目标

Agent 能周期性检查外部信息源，判断是否值得向用户主动发消息。

### 范围定义

1. 支持周期性 tick 执行。
2. 支持至少三类输入通道概念：
   - `alert`：高优先级事件
   - `content`：待筛选内容流
   - `context`：补充背景上下文
3. 支持基于用户规则和记忆进行推送判断。
4. 支持推送节流、冷却时间和基础频率控制。
5. 支持“本轮跳过”结果，不强制每轮产出消息。
6. 支持信息源配置与启停。
7. 推送来源统一走 `MCP` 抽象，避免为不同来源定制独立接入协议。

### 借鉴 Akashic 的点

1. 主动推送不是简单轮询转发，而是“拉取内容后由 Agent 判断是否发送”。
2. 将 `alert/content/context` 分层，避免所有输入混成单一队列。
3. 当没有可推内容时，将执行权交给 Drift。

### 一期不包含

1. 复杂配额模型
2. 高级多目标推送路由
3. 完整 Dashboard 调参界面

## 5.4 模块三：记忆系统

### 功能目标

将对话中的稳定事实、偏好、事件和近期上下文沉淀下来，并服务于回复、主动推送和 Drift。

### 范围定义

1. 建立 Markdown 记忆层，至少包含以下概念文件：
   - `MEMORY.md`：长期稳定事实
   - `SELF.md`：Agent 自我设定或关系设定
   - `HISTORY.md`：时间线事件
   - `RECENT_CONTEXT.md`：近期上下文摘要
   - `PENDING.md`：待归档事实缓冲
2. 建立向量记忆层，用于语义检索。
3. 每轮对话后支持 consolidation，将消息提取为事件和待归档事实。
4. 支持“记住这件事”“你记错了”两类显式记忆写操作。
5. 支持记忆去重、来源标记和纠错覆盖。
6. 支持定时或延迟式优化归档，避免频繁重写长期记忆。

### 借鉴 Akashic 的点

1. Markdown 层负责可读、可审计，向量层负责语义检索。
2. 通过 `PENDING.md` 作为缓冲层，降低长期记忆高频变化。
3. 近期上下文与长期记忆分开维护。
4. 向量记忆实现路线优先沿用 `akashic-agent` 的现有方法，降低一期探索成本。

### 一期不包含

1. 复杂记忆可视化后台
2. 全量记忆治理工具集
3. 自动化大规模回溯修复流程

## 5.5 模块四：Drift 空闲任务

### 功能目标

当系统当前没有值得主动推送的内容时，利用空闲周期执行受控后台任务。

### 范围定义

1. 当主动推送链路判定“无消息可发”时触发 Drift。
2. Drift 任务直接复用 `SKILL.md` 规范，避免重新设计任务描述格式。
3. 任务支持读写工作文件、读取记忆、查看消息历史、获取外部信息，并在必要时发送最多一条用户消息。
4. 每轮 Drift 必须显式结束并记录执行结果。
5. 支持最小间隔，防止频繁重复执行。

### 借鉴 Akashic 的点

1. Drift 是“可编排后台 Agent”，不是简单的 cron 脚本。
2. 任务定义文档化，让非核心代码也能扩展行为。
3. 推送动作要受限，避免 Drift 变成另一个无约束主动推送器。

### 一期建议预置任务

1. 记忆审计：抽检长期记忆与源消息是否一致
2. 画像补全：在合适时机生成轻量问题候选
3. 空闲整理：维护某个 backlog 或状态文件

## 5.6 模块五：多渠道接入能力

### 功能目标

通过统一的渠道适配层，使 `QQ Bot`、`Telegram Bot` 和后续 `Feishu Bot` 可以共享同一套 Agent 核心能力。

### 范围定义

1. 一期支持 `QQ Bot` 和 `Telegram Bot` 双通道接入。
2. 两个通道共享同一套会话处理、记忆、主动推送和 Drift 能力。
3. 渠道层负责消息收发、身份映射和会话上下文落点。
4. 为 `Feishu Bot` 预留扩展接口，但一期不要求完成正式接入。

### 一期不包含

1. 飞书正式上线接入
2. 多租户渠道隔离体系
3. 复杂渠道特性差异化能力

## 5.7 模块六：最小化开发调试 CLI

### 功能目标

为开发阶段提供一个轻量但可用的本地调试入口，帮助快速验证主链路，而不依赖真实机器人环境才能开发。

### 范围定义

1. 支持本地启动 Agent 主进程。
2. 支持本地发起一轮 CLI 对话，验证被动回复链路。
3. 支持查看基础状态，例如当前配置、工作区路径、记忆目录或调度状态。
4. 支持手动触发至少一轮主动推送 tick 或 Drift 验证。
5. 支持输出最小必要日志，便于排查消息处理和记忆流转问题。

### 一期不包含

1. 完整 TUI
2. 可视化调试界面
3. 高级运维命令集合

## 5.8 模块七：基础支撑能力

### 范围定义

1. 基础配置管理
2. 定时调度能力
3. 工作区目录约定
4. 决策日志与执行日志
5. 基础错误恢复与幂等处理

### 说明

这些能力不直接面向用户，但决定四个核心模块能否稳定联动，因此纳入一期范围。

## 6. 非目标

以下内容明确不作为本 PRD 一期交付目标：

1. 完整复刻 `akashic-agent` 的所有插件、工具和测试资产
2. 一开始就支持多平台、多租户、多用户隔离
3. 图形化 Dashboard、可视化检索面板、运维后台
4. 复杂推荐算法或强化学习式推送优化
5. 全自动、零配置的生产级部署方案
6. 一期完成 `Feishu Bot` 正式接入

## 7. 用户流程

### 7.1 被动回复流程

1. 用户发送消息
2. 系统读取会话上下文与长期记忆
3. 按需执行记忆检索和工具调用
4. Agent 生成回复
5. 系统提交本轮对话并触发记忆整理

### 7.2 主动推送流程

1. 调度器触发一次主动轮询
2. 系统从信息源拉取 `alert/content/context`
3. Agent 根据规则、记忆和上下文判断是否发送
4. 若值得发送，则生成并投递主动消息
5. 若不值得发送，则记录跳过原因
6. 若无可处理内容，则进入 Drift

### 7.3 记忆写入流程

1. 一轮对话结束后进入 consolidation
2. 系统抽取事件、稳定事实和近期上下文
3. 写入 `HISTORY.md`、`PENDING.md`、`RECENT_CONTEXT.md`
4. 定期将 `PENDING.md` 合并进 `MEMORY.md`
5. 向量层同步更新可检索内容

### 7.4 Drift 流程

1. 主动推送轮询为空
2. 系统选择可执行 Drift 任务
3. Agent 按任务说明执行后台操作
4. 如需触达用户，最多发一条消息
5. 记录结果并结束本轮

## 8. 验收标准

## 8.1 总体验收标准

以下条件全部满足，视为一期 PRD 对应能力具备可实现基础：

1. 四个核心模块均有清晰输入、处理、输出和状态落盘约定。
2. 任一模块关闭后，其余模块仍可独立工作，不互相阻塞。
3. 系统至少支持一条完整演示链路：
   - 被动对话产生记忆
   - 主动轮询读取规则和记忆
   - 无内容时进入 Drift
   - Drift 结果被记录

## 8.2 被动回复验收标准

1. 用户发送普通消息后，系统能在单轮内产出回复。
2. 回复前可读取长期记忆和近期上下文。
3. 用户显式要求“记住”后，记忆写入链路可被触发。
4. 用户指出“记错了”后，系统能走纠错路径而不是简单追加重复事实。
5. 一轮回复结束后，消息会进入后处理链路，为 consolidation 提供输入。
6. 同一套核心回复链路可被 `QQ Bot` 与 `Telegram Bot` 复用。

## 8.3 主动推送验收标准

1. 调度器可按配置周期触发主动轮询。
2. 系统能识别并处理 `alert/content/context` 三类输入。
3. 在存在候选内容时，系统可以做出“发送”或“跳过”的明确决策。
4. 主动消息受冷却时间或频率限制控制。
5. 当本轮没有可推送内容时，系统会进入 Drift，而不是空转或报错。
6. 所有主动轮询结果都有日志记录，至少可追溯“拉到了什么”“发没发”“为什么”。
7. 外部推送来源通过 `MCP` 抽象接入，而不是写死到单一来源实现中。

## 8.4 记忆系统验收标准

1. 记忆目录和核心文件结构清晰可落盘。
2. consolidation 至少能从一批对话中提取：
   - 一条或多条时间线事件
   - 一条或多条待归档事实
   - 一份近期上下文摘要
3. `MEMORY.md` 不会在每轮对话后都被直接重写。
4. 向量检索可根据语义召回相关记忆，用于被动回复或主动推送决策。
5. 同一批消息不会被重复写入记忆，或系统具备显式幂等保护。
6. 记忆内容具备来源追踪信息，便于后续审计和修正。
7. 向量记忆实现与 `akashic-agent` 路线保持兼容或高度相似，避免一期重造核心方案。

## 8.5 Drift 验收标准

1. 无主动推送内容时，系统可成功触发一轮 Drift。
2. Drift 至少支持从 `SKILL.md` 读取目标、步骤和工作文件。
3. Drift 能执行至少一种纯静默任务和一种可推送任务。
4. 一轮 Drift 结束时会记录：
   - 执行了哪个任务
   - 做了什么
   - 是否发消息
5. Drift 对用户的主动触达受限，单轮不会无限发消息。

## 8.6 多渠道接入验收标准

1. `QQ Bot` 和 `Telegram Bot` 可分别接入同一套 Agent 核心服务。
2. 两个渠道均可完成消息接收、回复发送和基础用户标识映射。
3. 主动推送能力可向两个渠道中的目标会话发出消息。
4. 渠道接入层不会侵入记忆、主动推送和 Drift 核心逻辑。
5. 为 `Feishu Bot` 预留的接口边界清晰，不需要大改核心模块即可追加实现。

## 8.7 最小化开发调试 CLI 验收标准

1. 开发者可通过 CLI 启动或连接本地 Agent 运行环境。
2. 开发者可通过 CLI 手动触发一轮被动对话验证。
3. 开发者可通过 CLI 手动触发一轮主动推送 tick 或 Drift。
4. CLI 至少能输出帮助信息和关键错误提示。
5. CLI 不依赖图形界面即可完成一期主要开发调试动作。

## 8.8 基础支撑验收标准

1. 所有核心配置项有明确存放位置和默认行为。
2. 核心工作目录结构稳定，便于后续扩展和排查。
3. 关键落盘数据具备最小可读性，不依赖专用工具才能排查。
4. 调度、主动推送、记忆归档、Drift 至少具备基础异常日志。

## 9. 建议实施顺序

建议按以下顺序推进实现：

1. 先落被动回复主链路和工作区结构
2. 再落记忆 consolidation 与召回
3. 然后接入主动推送调度与来源抽象
4. 最后补齐 Drift 任务框架

原因是记忆和上下文能力是另外三项能力的共享底座，而 Drift 依赖主动推送链路的“空闲判定”。

## 10. 里程碑建议

### M1：可对话

1. 能接收消息并生成回复
2. 有基础工作区和日志

### M2：可记忆

1. 能写入和检索长期记忆
2. 有 consolidation 和近期上下文

### M3：会主动

1. 能轮询信息源并做推送决策
2. 有节流和跳过逻辑

### M4：多渠道与调试闭环

1. `QQ Bot` 和 `Telegram Bot` 打通
2. 最小化开发调试 CLI 可用

### M5：会空闲自驱

1. 无推送内容时能跑 Drift
2. 至少有 2 到 3 个可运行的预置任务

## 11. 打开问题

以下问题建议在架构设计阶段明确：
